# OpenChrome v1.10.3 (staging)

Release date: TBD

## Summary

v1.10.3 is a small **performance / resource-usage patch release**. The
`serve` action previously started a `/health` + `/metrics` HTTP listener
in every transport mode, including `stdio`. Stdio-launched instances
(1 per MCP client, typically 60-80 concurrent on an active developer
machine) paid the cost of that listener for no operational benefit,
because stdio clients talk to openchrome over the pipe, not over HTTP.
v1.10.3 gates the listener behind transport mode + an explicit env var.

Tracks [issue #648](https://github.com/shaun0927/openchrome/issues/648).

---

## Resource usage

- Stdio-mode `serve` no longer binds the health port by default. That
  removes one listening TCP socket (one file descriptor + `http.Server`
  + per-listener event wiring) from each stdio instance, saving roughly
  200-300 KB heap per process. With 60-80 concurrent stdio instances
  typical of a heavily-used dev machine, the cumulative saving is in
  the 12-24 MB range, plus 60-80 fewer kernel TCP sockets.
- Daemon-mode deployments (`--transport http` or `--transport both`)
  are unchanged: `/health` and `/metrics` remain bound exactly as in
  v1.10.2. Operators running daemon-mode health checks externally can
  opt out with `OPENCHROME_HEALTH_ENDPOINT=0`, and operators running
  stdio under a supervisor who still want the listener can opt in
  with `OPENCHROME_HEALTH_ENDPOINT=1`.
- One deterministic log line per startup (`[SelfHealing]
  HealthEndpoint: enabled ...` / `... disabled ...`) declares the
  resolved state, so every instance records what it decided on stderr.

---

## Configuration

| Env var | Default | Description |
|---|---|---|
| `OPENCHROME_HEALTH_ENDPOINT` | unset | `1` / `true` forces the health endpoint on; `0` / `false` forces it off; anything else falls through to the transport-mode default (on for `http`/`both`, off for `stdio`). |

No CLI flag changes. No new dependencies. No port or payload changes.

---

## Compatibility notes

The only externally observable difference is that stdio-mode `serve`
no longer binds the health port by default. If an internal tool was
relying on that listener in stdio mode, re-enable it explicitly with
`OPENCHROME_HEALTH_ENDPOINT=1`. The recommended long-term path is to
run daemon-mode (`--transport http` / `both`), where the listener
remains on by default.

The teardown path was updated at the same time: `healthEndpoint?.stop()`
replaces `healthEndpoint.stop()` in `enhancedShutdown` so that the
null-in-stdio case cannot raise `TypeError: Cannot read properties of
null` during SIGTERM / SIGINT / SIGHUP.

## Verification performed for v1.10.3

- new unit suite: `tests/utils/health-endpoint-gating.test.ts`
  (table-driven coverage of all 8 resolver cases from issue #648 §5.2).
- new integration suite: `tests/integration/health-endpoint-gating.test.ts`
  (spawns `node dist/index.js serve ...` in seven configurations,
  probes the health port with `net.Socket.connect()` + short timeout —
  no errno-string matching, so Windows and POSIX share the test path —
  and asserts each scenario's graceful-shutdown exit code is 0 with no
  `TypeError` / `Cannot read properties of null` / `UnhandledPromiseRejection`
  on stderr).
- benchmark script: `scripts/bench-health-endpoint.mjs` (spawns 20
  instances per configuration, records RSS + FD count, medians the
  result).
- existing test matrix passes unchanged.
